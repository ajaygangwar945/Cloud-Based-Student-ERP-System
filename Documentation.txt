================================================================================
          PRODUCTION SYSTEM DOCUMENTATION: CLOUD-BASED STUDENT ERP SYSTEM
================================================================================

This document serves as the official, comprehensive project manual and technical
specification for the Cloud-Based Student ERP System. It details the actual 
production implementation, architectural components, relational database schema,
container configurations, deployment blueprints, CI/CD automated pipeline, and 
observability integrations.

--------------------------------------------------------------------------------
1. EXECUTIVE SUMMARY & TECH STACK
--------------------------------------------------------------------------------
The legacy student records portal has been fully re-engineered into a modern 3-tier,
highly-available, scalable, and observable cloud application. It utilizes modern
DevOps automation and GitOps principles to deploy seamlessly across virtual nodes.

Active Technology Stack:
* Frontend: Vite + React 19, Axios, dynamic role-based dashboards, CSS dark mode, 
  glassmorphism UI styling, and responsive tables.
* Backend API: Node.js + Express, MySQL2 relational driver, JWT authentication, 
  Bcrypt secure password hashing, and prom-client telemetry.
* Relational Database: MySQL Relational Database Service.
* Containerization: Docker (multi-stage production builds and Docker Compose).
* Orchestration: Multi-Node Kubernetes (control plane and workers running Calico CNI).
* IaC: Terraform provisioning AWS EC2 instances, security firewalls, and networks.
* Automated CI/CD: Jenkins pipeline (Declarative pipeline as code).
* Observability Stack: Prometheus (time-series scraping) and Grafana (metrics dashboard).

3. RELATIONAL DATABASE SCHEMA (MySQL)
--------------------------------------------------------------------------------
The institutional data layer is constructed inside the 'student_erp' database 
defined in `database/init.sql`. The schema enforces referential integrity:

* 'users': Core account details.
  - Column: id (INT Auto-Increment, Primary Key)
  - Column: username (VARCHAR, Unique)
  - Column: password (VARCHAR, Salted Bcrypt Hash)
  - Column: role (ENUM: 'admin', 'student', 'faculty')
* 'departments': Institutional divisions.
  - Column: id (INT Auto-Increment, Primary Key)
  - Column: name (VARCHAR, Unique)
* 'students': Profile registry linked to users & departments.
  - Column: id (INT Auto-Increment, Primary Key)
  - Column: user_id (INT, Foreign Key referencing users.id ON DELETE CASCADE)
  - Column: department_id (INT, Foreign Key referencing departments.id ON DELETE SET NULL)
  - Column: enrollment_no (VARCHAR, Unique)
  - Column: semester (INT)
* 'faculty': Teacher profiles.
  - Column: id (INT Auto-Increment, Primary Key)
  - Column: user_id (INT, Foreign Key referencing users.id ON DELETE CASCADE)
  - Column: department_id (INT, Foreign Key referencing departments.id ON DELETE SET NULL)
  - Column: designation (VARCHAR)
* 'attendance': Daily logs with subjects and states.
  - Column: id (INT Auto-Increment, Primary Key)
  - Column: student_id (INT, Foreign Key referencing students.id ON DELETE CASCADE)
  - Column: subject (VARCHAR)
  - Column: attendance_date (DATE)
  - Column: status (ENUM: 'present', 'absent', 'late')
* 'marks': Course grades linked to students.
  - Column: id (INT Auto-Increment, Primary Key)
  - Column: student_id (INT, Foreign Key referencing students.id ON DELETE CASCADE)
  - Column: subject (VARCHAR)
  - Column: marks_obtained (DECIMAL(5,2))
  - Column: max_marks (DECIMAL(5,2))
  - Column: semester (INT)
* 'fees': Invoicing ledger.
  - Column: id (INT Auto-Increment, Primary Key)
  - Column: student_id (INT, Foreign Key referencing students.id ON DELETE CASCADE)
  - Column: amount_due (DECIMAL(10,2))
  - Column: amount_paid (DECIMAL(10,2))
  - Column: status (ENUM: 'paid', 'pending', 'partial')
  - Column: due_date (DATE)
* 'notices': Broadcast announcements.
  - Column: id (INT Auto-Increment, Primary Key)
  - Column: title (VARCHAR)
  - Column: content (TEXT)
  - Column: posted_by (INT, Foreign Key referencing users.id ON DELETE SET NULL)

4. CONTAINERIZATION & MULTI-STAGE OPTIMIZATION (Docker)
--------------------------------------------------------------------------------
To guarantee clean builds and lightweight virtual environments, optimized Docker 
configurations are implemented.

A. Backend Dockerfile (`backend/Dockerfile`):
Uses a single-stage, lightweight `node:20-alpine` base image, locking down 
development configurations and injecting production variables.
* Directory: `/app`
* Exposes: Port `5000`

B. Frontend Dockerfile (`frontend/Dockerfile`):
Adopts a highly-efficient **Multi-Stage Build** to compile static assets and
serve them through Nginx:
* **Stage 1 (Build):** Compiles the Vite React code inside a `node:20-alpine` 
  container. Emits highly-optimized Javascript/CSS bundles into `/app/dist`.
* **Stage 2 (Nginx):** Copies the built assets from Stage 1 directly into an 
  ultra-slim `nginx:alpine` image. Static assets are stored inside Nginx's 
  HTML root directory (`/usr/share/nginx/html`).
* **Result:** Production size is reduced from 450MB+ down to ~35MB, minimizing
  disk requirements and accelerating page load speeds.

5. KUBERNETES ORCHESTRATION BLUEPRINT
--------------------------------------------------------------------------------
The production Kubernetes manifests deploy the application onto multi-node clusters:

* **MySQL Deployment (`k8s/mysql-deployment.yaml`):**
  Spins up a persistent database container. Mounts storage path `/var/lib/mysql`
  directly inside the pod to prevent data loss. Configured via `mysql-configmap.yaml`
  to securely inject database name (`student_erp`), port (`3306`), and users.
* **Backend API Deployment (`k8s/backend-deployment.yaml`):**
  Spins up Node.js pods running Express. Exposes endpoints to frontend clients 
  using a Kubernetes Service exposed on NodePort `30001`.
* **Frontend Web Deployment (`k8s/frontend-deployment.yaml`):**
  Runs the compiled React/Nginx containers. Renders high-speed, dynamic portals.
  Exposed to the public internet using a Kubernetes Service mapped to NodePort `30000`.

6. AUTOMATED GITOPS CI/CD PIPELINE (Jenkins)
--------------------------------------------------------------------------------
The Jenkins pipeline is written as a declarative pipeline-as-code inside the root 
`Jenkinsfile`. When a developer pushes code to GitHub, webhooks automatically 
trigger the following continuous integration and rollout stages:

1. **Pull Code:** Pulls the active repository branch (`main`) dynamically.
2. **Run Tests:** Triggers validation, integration, and linting checks.
3. **Docker Hub Login:** Authenticates against the Docker Registry using credentials 
   stored securely in Jenkins Credentials Manager (`docker-hub-credentials`).
4. **Build Docker Images:** Compiles both frontend and backend Dockerfiles. Tags 
   them with two tags: `latest` and a unique `${BUILD_NUMBER}` for strict build auditing.
5. **Push to Docker Hub:** Uploads the built images to Docker Hub (`ajaygangwar945/`).
6. **Deploy to Kubernetes:** 
   - Applies the K8s configurations (`kubectl apply -f k8s/`).
   - Performs a zero-downtime rolling update rollout by explicitly targeting 
     the fresh build tag:
     * `kubectl set image deployment/backend backend=ajaygangwar945/student-erp-backend:${BUILD_NUMBER}`
     * `kubectl set image deployment/frontend frontend=ajaygangwar945/student-erp-frontend:${BUILD_NUMBER}`
   - Tracks rollout status to confirm container readiness (`kubectl rollout status`).
7. **Cleanup (Post Action):** Deletes local images built in the workspace to save 
   precious virtual hard drive storage on the AWS master node.

7. MONITORING & OBSERVABILITY INTEGRATIONS (Prometheus & Grafana)
--------------------------------------------------------------------------------
Real-time monitoring tells us exactly how our live cloud cluster is behaving:

* **Telemetry Scraping:**
  The Express API leverages `prom-client` to capture active endpoint volumes, 
  latencies, CPU pressure, RAM usage, and active mock traffic spikes. These metrics
  are exposed to the internal network on `/metrics`.
* **Prometheus Setup:**
  Prometheus crawls all cluster pods and scrapes `/metrics` endpoints. Configured via 
  `prometheus-rbac.yaml` with necessary service account permissions. Mapped to 
  NodePort `30090`.
* **Grafana Dashboards:**
  Grafana connects directly to Prometheus. Exposes premium metrics dashboards mapped 
  to NodePort `30030`.
* **Persistence Fix:**
  Prometheus and Grafana deployments are pinned to Worker Node 1 using Kubernetes 
  `nodeSelector`. Their data directories are mapped to host directory paths 
  (`/var/lib/prometheus-storage` and `/var/lib/grafana-storage`). This preserves 
  scraped metrics, data pipelines, and dashboards across reboots or VM power cycles!

8. SECURITY ARCHITECTURE
--------------------------------------------------------------------------------
The Student ERP security follows a robust, modern defense-in-depth model:
* **Password Hashing:** Passwords are never stored in plaintext. They are salted 
  and hashed securely using **Bcrypt** (60-character hash length).
* **Role-Based Access Control (RBAC):** Users are assigned a strict, immutable 
  role ('admin', 'student', or 'faculty'). The API verifies user roles inside JWTs 
  to protect restricted operations.
* **JSON Web Token (JWT):** Secure token authentication. The backend signs a JWT 
  upon login using a secure signature secret. The token is valid for 2 hours, 
  allowing students and teachers to browse safely.
* **Securing secrets:** Host systems inject database passwords via Kubernetes 
  ConfigMaps rather than hardcoding them in code.

9. PRODUCTION CLOUD LIFE-CYCLE OPERATIONS
--------------------------------------------------------------------------------
To manage your production AWS EC2 multi-node cluster, keep these commands on hand:

1. Waking up all cloud instances:
   > aws ec2 start-instances --instance-ids i-0a28f67221289af6a i-05d75c5b835130230 i-08624e36ec8a4e35d

2. Refreshing dynamic IP mapping:
   > cd terraform
   > terraform refresh
   > cd ..

3. Restoring internal CNI bridge routing on reboots:
   > ssh -i terraform/erp-cloud-key.pem ubuntu@<MASTER_IP> "sudo modprobe br_netfilter && sudo sysctl net.bridge.bridge-nf-call-iptables=1"
   > ssh -i terraform/erp-cloud-key.pem ubuntu@<WORKER_1_IP> "sudo modprobe br_netfilter && sudo sysctl net.bridge.bridge-nf-call-iptables=1"
   > ssh -i terraform/erp-cloud-key.pem ubuntu@<WORKER_2_IP> "sudo modprobe br_netfilter && sudo sysctl net.bridge.bridge-nf-call-iptables=1"

4. Clean pod rollout restarts:
   > ssh -i terraform/erp-cloud-key.pem ubuntu@<MASTER_IP> "kubectl rollout restart deployment backend frontend mysql prometheus grafana"

5. Verification:
   > ssh -i terraform/erp-cloud-key.pem ubuntu@<MASTER_IP> "kubectl get pods -o wide"
